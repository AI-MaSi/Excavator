Firmware changes now in place in this repo:

  Wiring note (confirmed on hardware, keep this fixed):

  - Board: Seeed XIAO RP2040
  - I2C0 bus uses P28/P29
  - I2C1 bus uses P6/P7
  - P26/P27 does NOT have the second IMU bus on this hardware setup
  - Do not change these pin assignments unless the physical wiring is changed too

  - IMU discovery is runtime-based, not fixed-slot.
  - Probes all four valid locations:
      - I2C0 @ 0x6A
      - I2C0 @ 0x6B
      - I2C1 @ 0x6A
      - I2C1 @ 0x6B
  - Detection uses WHO_AM_I == ISM330DHCX_ID.
  - Active IMU list is cached after init and reused until restart/re-init.
  - Data frame sensor count is variable and matches detected IMUs.
  - Descriptor frames are sent at stream startup and retried a few times.
  - Descriptor payload includes:
      - sensor count
      - sample rate
      - per-sensor bus
      - per-sensor addr
  - Tilt extraction was changed to gravity-based pitch/roll, not 2*atan2(y,w).
  - Firmware-side zero/identity command handling was removed as unused.

  Startup calibration improvement: auto stationary check

  Goal:

  - Favor better gyro bias estimation over fast boot.
  - Reduce user-facing tuning knobs; keep calibration acceptance logic in firmware.
  - Base acceptance thresholds on measured machine data, not guessed values.

  Implemented firmware behavior:

  - Separate startup settle phase removed.
  - Startup gyro-bias measurement window increased to 30 s.
  - During that 30 s window, continuously collects per-IMU gyro and accel statistics:
      - sample count
      - gyro mean (used as bias)
      - gyro stddev per axis
      - accel mean per axis
      - accel stddev per axis
      - accel norm mean/min/max
      - read error count
  - Sends CAL_WAIT during the full measurement period so the host shows
    that calibration is still running.
  - At the end of the 30 s window:
      - if stats pass the stationary test, apply the measured gyro mean as bias
      - report measured stats to the host in the startup handshake
      - if stats fail, stay in ERR_CAL as today
  - Removed the unused user-facing offset_s / OFFSET_S setting from the host,
    config file, and Pico settings parser.

  Proposed data collection / tuning process:

  - Use the host-side boot-time logging that prints one concise calibration report per IMU.
  - Collect several clean power-up runs with the machine intentionally left fully still.
  - Collect a few disturbed runs with small bumps / vibration so we can see what bad
    calibration windows look like.
  - Save the reported stats from both groups and compare:
      - worst good-case gyro stddev
      - worst good-case accel stddev
      - accel norm range during good cases
      - separation margin versus disturbed cases

  Threshold-setting rule:

  - Set each stationary threshold just above the worst clean-run value, with some margin.
  - Verify that those thresholds still reject the disturbed runs.
  - Keep accel-norm bounds broad unless real data shows they should be tightened.
  - Do not expose these thresholds as normal user config unless a real field need appears.

  Expected outcome:

  - Better startup gyro bias from a longer measurement window.
  - Less reliance on guessed hard-coded values.
  - Fewer user-visible settings; calibration behavior becomes mostly automatic.
